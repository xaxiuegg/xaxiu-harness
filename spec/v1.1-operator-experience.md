## 1. Operator Interaction Surface Map

Every UI moment the operator sees, from install through daily use. For each moment: what they see, what they do, what the harness does under the hood.

| Moment | Operator sees | Operator does | Harness does (invisible) |
|---|---|---|---|
| **Install** | Double‑click `xaxiu-harness-setup-v1.0.exe` → Inno Setup wizard (destination folder, Start Menu folder, “Allow Task Scheduler” checkbox). | Click Next, Accept, Install. May need admin elevation. | Bundles Python 3.11+ runtime, pip‑installs `xaxiu-harness` package, writes `schtasks` entries, creates `state/` directory. |
| **First Run (post‑install)** | PyWebView window: “Welcome to xaxiu Harness”, then step‑by‑step “API Key” entry (pre‑filled from `SET` if detected), “Optional: import xaxiu‑swarm config”, “Set up first project?” template picker. | Enter keys, pick template (e.g. `warehouse-style`), name project, click Finish. | Writes `harness.config.yml` (version, default project), writes project’s `adapter.yaml`, runs `harness install` to schedule tasks, opens browser to `localhost:7878`. |
| **Daily – Dashboard** | Browser at `localhost:7878` – dark steel background, animated pipeline, engine tiles (pulse colour / health), event stream, priority toggles, project selector. | Monitor, toggle priorities, view dispatch history, click on a dispatch to see decision archaeology. | FastAPI serves static assets, WebSocket pushes real‑time status, reads `state/engine_health.json`, `state/active_dispatches.json`, `state/history.db`. |
| **Add new project** | “+” button → template picker (5 templates), project name, project root. After creation: form‑based YAML editor opens for customisation. | Pick template, name project, set root folder, optionally tweak in form editor. | Copies template YAML to `adapters/<project>/harness-adapter.yaml`, creates project entry in `state`, updates dashboard. |
| **Edit project config** | Dashboard → Project Settings → Visual Config Builder (Section 3). Forms for routing rules, status backend, observer, scheduled tasks. | Fill forms, click Save. May use Natural‑Language translator (Section 4) to generate initial config. | Every save writes the entire adapter YAML file atomically. Validates against Pydantic schema; shows error on failure. |
| **Power‑user fallback** | “Edit YAML directly” button in project settings → a textarea with syntax‑highlighted YAML, “Revert to last form state” and “Save YAML” buttons. | Edit raw YAML, click Save. Harness validates on save; if invalid, shows diff and rejects. | Writes YAML only after passing Pydantic schema. Updates visual builder to reflect changes. |

## 2. Template Library (≥5 Starter Templates)

Each template is a complete `harness-adapter.yaml` plus a one‑paragraph operator‑facing description shown in the picker.

### 2.1 `warehouse-style`

```yaml
name: warehouse-style
project_root: "{{PROJECT_ROOT}}"
status_tracking:
  backend: csv
  config:
    csv_path: STATUS.csv
    headers: [date, task, engine, status, notes]
routing_rules:
  - if: "packet*.md"
    then:
      backend: deepseek
      model: deepseek-v4-flash
      extra_args: { "--no-thinking": true }
    reason: "Default for warehouse – patches need anchor precision"
  - if: "*-screenshots-*"
    then:
      backend: kimi
      model: kimi-v2
    reason: "Only Kimi handles screenshot extraction"
observer:
  enabled: true
  cadence_minutes: 30
  daily_retro_time: "17:00"
  flag_patterns: [".*FAIL.*", ".*BLOCKER.*"]
scheduled_tasks:
  - cron: "0 9 * * 1-5"
    command: "harness observer-tick --project warehouse-style"
    idempotent: true
  - cron: "0 17 * * 1-5"
    command: "harness retro --project warehouse-style"
    idempotent: false
  - cron: "*/15 * * * *"
    command: "harness status --project warehouse-style --report"
    idempotent: true
```

**Description:** Ideal for teams that need continuous drift detection, daily retro, and CSV‑based status logs. Includes observer cycles and screenshot‑specific Kimi routing. Default for warehouse‑like workflows.

### 2.2 `generic-coding`

```yaml
name: generic-coding
project_root: "{{PROJECT_ROOT}}"
status_tracking:
  backend: markdown
  config:
    path: STATUS.md
    headers: [date, task, engine, status, notes]
routing_rules:
  - if: "*.py"
    then:
      backend: kimi
      model: kimi-v2
    reason: "Code generation preferred on Kimi"
  - if: "*.md"
    then:
      backend: deepseek
      model: deepseek-v4-flash
    reason: "Large files use DeepSeek"
observer:
  enabled: false
scheduled_tasks:
  - cron: "0 0 * * *"
    command: "harness status --report --project generic-coding"
    idempotent: true
```

**Description:** Simple coding assistant with Kimi for Python, DeepSeek for markdown. No observer. Status written as a Markdown table. Good for solitary dev or small team.

### 2.3 `writing-content`

```yaml
name: writing-content
project_root: "{{PROJECT_ROOT}}"
status_tracking:
  backend: markdown
  config:
    path: STATUS.md
    headers: [date, task, engine, status, words]
routing_rules:
  - if: "*-draft.md"
    then:
      backend: kimi
      model: kimi-v2
    reason: "Kimi for first drafts"
  - if: "*-edit.md"
    then:
      backend: deepseek
      model: deepseek-v4-flash
    reason: "DeepSeek for polishing and cutting"
observer:
  enabled: false
scheduled_tasks: []
```

**Description:** For writers and content creators. Drafts go to Kimi, editing/polishing to DeepSeek. Status table tracks word counts. No scheduled tasks – manual dispatches only.

### 2.4 `research-comparison`

```yaml
name: research-comparison
project_root: "{{PROJECT_ROOT}}"
status_tracking:
  backend: csv
  config:
    csv_path: COMPARISON.csv
    headers: [date, packet, engine_a, engine_b, result_a, result_b]
routing_rules:
  - if: "*"
    then:
      backend: burst
      extra_args: { engines: [kimi, deepseek] }
    reason: "Route to both engines for A/B comparison"
observer:
  enabled: false
scheduled_tasks: []
```

**Description:** Routes every packet to two engines (Kimi and DeepSeek) for side‑by‑side comparison. Results stored in a CSV with columns per engine. No observer. Prefix `burst` indicates special multi‑engine dispatcher.

### 2.5 `solo-dev`

```yaml
name: solo-dev
project_root: "{{PROJECT_ROOT}}"
status_tracking:
  backend: csv
  config:
    csv_path: STATUS.csv
    headers: [date, task, result, notes]
routing_rules:
  - if: "*"
    then:
      backend: deepseek
      model: deepseek-v4-flash
      extra_args: {}
    reason: "Single engine for simplicity"
observer:
  enabled: true
  cadence_minutes: 60
  daily_retro_time: "18:00"
  flag_patterns: [".*ERROR.*"]
scheduled_tasks:
  - cron: "0 18 * * *"
    command: "harness retro --project solo-dev"
    idempotent: false
```

**Description:** Minimal configuration for a single developer with one engine. Observer runs every hour and daily retro at 18:00. No complex routing – just send everything to DeepSeek.

## 4. Natural‑Language → YAML Translator

Operator types into the dashboard: “Set up my book-writing project. Drafts to Kimi, polishing to DeepSeek, status as a markdown file.”  
The harness calls a built‑in low‑cost LLM (Kimi‑api) to produce valid adapter YAML, shows it in the visual builder, operator tweaks via forms, saves.

### 4.1 Prompt Template

```
You are an expert in converting natural-language descriptions of a “harness” project configuration into valid YAML.
The YAML must follow this Pydantic schema (fields omitted for brevity, but must be exact):
- `name`: string, required
- `project_root`: string, required
- `routing_rules`: array of objects with `if`, `then` (object with `backend`, `model`, optional `extra_args`), and optional `reason`.
- `status_tracking`: object with `backend` (one of `csv`, `markdown`, `jira`, `linear`) and `config` object.
- `observer`: object with `enabled`, `cadence_minutes`, `daily_retro_time`, `flag_patterns`.
- `scheduled_tasks`: array of objects with `cron`, `command`, `idempotent`.

User input: "{{USER_INPUT}}"

Return ONLY the YAML block, no explanation. If the input is ambiguous, make sensible defaults (e.g. observer enabled cadence 30, no scheduled tasks). Use {{PROJECT_ROOT}} as placeholder.
```

### 4.2 Validation Pipeline

1. LLM returns YAML string.
2. Parse with `yaml.safe_load`.
3. Validate against `AdapterConfig` Pydantic model (from `src/harness/adapters/schema.py`).
4. If validation fails → show error message in a toast: “The translation did not produce a valid configuration. Error: [field] [error]. Would you like to refine your description?”
5. If valid → populate the visual builder forms with the parsed data, allowing operator to review / adjust before saving.
6. On “Save” from form → re‑serialize to YAML and write file.

### 4.3 Error‑Handling UX

- When Pydantic validation fails, the toast includes a human‑readable version of the error (e.g. “Missing required field: ‘status_tracking.backend’. Options: csv, markdown, jira, linear.”)
- Operator can either re‑type a refined description or click “Edit YAML Directly” to fix manually.
- The LLM call is synchronous (within dashboard request, <5s). A loading spinner appears.

## 5. Windows Installer (Inno Setup) Full Flow

### 5.1 Setup Wizard (Double‑click)

- **Welcome page** – product name, version, link to docs.
- **Destination location** – default `C:\Program Files\xaxiu-harness` (browse).
- **Start Menu folder** – default `xaxiu Harness`.
- **Optional components** – checkbox: “Install Task Scheduler tasks (requires admin)”. Unless checked, manual install later needed via `harness install`.
- **Ready to install** – summary.
- **Install** – copies bundled Python (embedded `python` directory, ~60MB), pip‑installs `xaxiu-harness` from embedded wheel, writes `harness.config.yml` to `%APPDATA%\xaxiu-harness\state\`, adds shortcuts.
- **Post‑install** – if “Launch first‑run wizard” checked (default), runs `first-run-wizard.exe` (PyWebView). Otherwise, user can launch later from Start Menu.

### 5.2 First‑Run Wizard (PyWebView)

```
Window title: xaxiu Harness Setup
Size: 800×600, centered.

Pages (HTML rendered in PyWebView):
1. Welcome    – logo, “Welcome to xaxiu Harness v1.0”
2. API Keys   – fields for DEEPSEEK_API_KEY, KIMI_API_KEY, ANTHROPIC_API_KEY.
               Each field has a toggle visibility (eye icon).
               If any key is already in environment variables, show “✓ Detected”.
3. xaxiu‑swarm import – checkbox: “Import existing xaxiu‑swarm config?” (scans common locations).
               If found → preview adapters to import; user selects which.
4. First project – “Set up a new project?”
               Template picker (5 templates with descriptions).
               Project name, project root (browse folder).
5. Finish      – “Open Dashboard” checkbox (default).
               Click “Finish” → close wizard, launch browser to localhost:7878.

Cancelling at any point leaves harness installed but without any project.
```

### 5.3 Silent Operation

After first run, harness runs entirely through Task Scheduler and dashboard. No further UI popups.

### 5.4 Uninstaller

- Inno Setup provides uninstall entry in “Add/Remove Programs”.
- On uninstall: runs `harness install --uninstall` (removes all `schtasks`), deletes `state/` directory, removes bundled Python.
- Does **not** delete `adapters/` or user files by default (offers checkbox “Remove all project data” during uninstall).

### 5.5 Upgrade Installer (In‑Place)

- Running the installer over an existing installation detects `harness.config.yml`.
- Offers “Upgrade” mode: preserve `state/` except `engine_health.json` (reset health), preserve `adapters/`, update Python runtime if newer.
- After upgrade, runs `harness install` to update tasks and then opens dashboard.

## 6. Satisfactory‑Inspired Dashboard Aesthetic

### 6.1 Visual Direction

| Property | Value |
|---|---|
| **Base background** | `#0F1318` (very dark steel) |
| **Background texture** | Subtle blueprint grid (1px `#1A212A` lines, 20px spacing) |
| **Accent colours** | Amber `#FFA94D` (warnings/status), Cyan `#4DD0E1` (active queries, pipes) |
| **Text** | `#E0E0E0`, headings `#FFFFFF` |
| **Code/monospace** | `JetBrains Mono`, size 14px |
| **Status pills** | ● (green) running, ○ (grey) idle, ✓ (cyan) success, ✗ (red) failure, ⚠ (amber) warning, ⟲ (cyan) fallback |
| **Conveyor flow** | Animated dotted line (2px, dash‑array, moving) from project → engine → dispatch |
| **Engine tiles** | Rounded rects, slight 3D edge, colour shifts based on load: idle (`#1A212A`) → busy (`#FFA94D` 20%) → overloaded (`#FF5252` 40%) |
| **Flash/transitions** | Failure pulse red (`#FF5252`) 300ms, recovery pulse green (`#4CAF50`) 500ms |
| **Decision archaeology panel** | Slide‑out from right when dispatch clicked: shows packet content, fallback chain as connected dots, patch diff, latency/ cost, result status |

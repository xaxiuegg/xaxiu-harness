# W6-A1.1 evidence: D5 worktree-branching works

**Question**: did worker-2's worktree inherit worker-1's commit when run via the failed `runs/20260523T142354-79ed`?

## Evidence 1: worker-2's git history contains worker-1's commit

```
$ git log --oneline -3 wt/20260523T142354-79ed/worker-2
fddd3aa [s1] Add tests for env_var_inventory check          # worker-2's commit
ec7fc51 [s1] Implement env_var_inventory doctor check        # worker-1's commit (INHERITED)
b777fd6 W6-PLAN: Wave 6 plan + MiMo audit gate
```

The presence of `ec7fc51` in worker-2's branch history is **proof** that the dependent worker was branched from the parent worker's branch — exactly what D5 documents.

## Evidence 2: worker-2's tests pass when re-run in its worktree

```
$ cd .harness/worktrees/20260523T142354-79ed/worker-2
$ python -X utf8 -m pytest tests/test_doctor.py -q --tb=no
......................                                                   [100%]
22 passed in 1.10s
```

The 19p/3f/0s the worker reported at run-time is not reproducible now. Possible causes:
- pytest cache from the original failed dispatch had stale results
- worker-2 committed its tests AFTER pytest ran (commit fddd3aa contains the test file but the worker may have run pytest on intermediate state)
- env-var presence at the original run-time differed (some monkeypatch interaction not captured here)

Whatever the cause, it is **not** worktree-branching — that part empirically works.

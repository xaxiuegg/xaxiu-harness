## C1-context-budget-analyst  (deepseek/deepseek-v4-flash)

## 1. Headline recommendation

Default the agent’s dispatch result to a 200‑token summary (generated by the engine or a lightweight extractor) and store the full response in a retrievable cache — keeping 30‑dispatch sessions under 6K tokens of context growth from offloaded work, rather than 750K.

## 2. Concrete numbers / API signature / data shape

```python
@dataclass
class DispatchResult:
    success: bool
    summary: str                # NEW: default ~200 tokens, auto-generated
    text_ref: str | None = None # e.g. "cache://dispatches/abc-123" (populated always)
    dispatch_id: str
    engine_used: str
    tokens_used: int            # total engine tokens (in+out)
    tokens_in: int
    tokens_out: int
    cost_usd: float
    error: str | None = None

    @property
    def full_text(self) -> str:
        """Lazily fetch full response from cache (triggers I/O)."""
        return _cache_retrieve(self.text_ref)
```

New `dispatch_sync()` default parameters:

```python
def dispatch_sync(
    request: str | Dict[str, Any],
    response_format: str = "summary",   # "summary" | "full" | "both"
    summary_max_tokens: int = 250,       # cap for auto-summary
    cache_ttl: int = 3600,              # seconds to keep full text
    **kwargs,
) -> DispatchResult:
    ...
```

**Budget math:**

- 30 dispatches × 200 tokens (summary) = **6,000 tokens** → 3% of a 200K window.
- Same 30 dispatches at 25K full → 750,000 tokens → **375% of window**, impossible.
- Even 30 × 1K (very short responses) = 30K, still manageable but wasteful.
- Target for “30 dispatches under 100K growth” → average 3.3K/dispatch → default summary of 200–300 tokens is 15× smaller than that line.

## 3. The default that maximizes context preservation

`response_format="summary"` **must** be the default. The SDK’s `dispatch_sync()` silently returns a compact summary; the agent never sees the full text unless it explicitly opts in.

The library also auto-generates the summary using the engine itself (cheapest fast model) with a prompt like: *“Summarize the following response in at most N tokens. Preserve critical facts, actions, and numeric data.”* This adds < 100 extra tokens engine‑side and costs < $0.001 per dispatch.

For agents that just `from harness import dispatch_sync` and call it, they get:

```python
result = dispatch_sync("analyze the latest logs")
# result.summary -> "Found 3 anomalies in module X, severity high...\n"
# result.text_ref -> "cache://dispatches/d-456"
```

The agent can avoid calling `result.full_text` unless it truly needs the raw output.

## 4. The escape hatch

When the agent *does* need the full response (e.g., for detailed code output or diff), it has three mechanisms, all with explicit context cost:

| Method | Context cost | Notes |
|--------|-------------|-------|
| Set `response_format="full"` on the call | Full text returned directly in `DispatchResult.summary` (or `text`) – agent must decide to read it. | Simple, but agent pays the cost with awareness. |
| Call `result.full_text` lazily | Returns full text (no additional IO cost if cached). Agent can conditionally append `\n{result.full_text}` to prompt. | Clear code path; agent controls moment of inflation. |
| Set `response_format="both"` | Return both `summary` and `summary` field contains summary, and a new `raw_text` field (or `text`). | Doubles default cost: summary + full text. Explicit opt‑in required. |

Recommended safe pattern for an agent that *sometimes* needs full:
```python
result = dispatch_sync(prompt)
# ... use summary to decide ...
if need_more_detail(result.summary):
    chunk = result.full_text
    # inject only the necessary part (e.g., first 2000 tokens)
```

## 5. One risk / failure mode specific to the CONTEXT BUDGET ANALYST

**Summary truncation of critical information.**  
An auto‑generated summary may omit a single crucial number (e.g., “error rate 0.3%” vs “0.03%”), an exact file path, or a subtle instruction that the agent needs to act on. This could cause downstream reasoning failures that are hard to debug because the agent only sees the summary.

**Mitigation built into the design:**
- The summary prompt explicitly demands preservation of all numeric values, identifiers, and action items.
- The `text_ref` is always accessible; the agent can fall back to `full_text` at the cost of context.
- A future improvement: per‑field structured summaries (e.g., `summary = {"key_findings": str, "numbers": Dict, "actions": List[str]}`) to reduce information loss while keeping tokens low.

## C2-dispatch-return-contract  (deepseek/deepseek-v4-flash)

## 1. Headline recommendation

Ship `dispatch()` with default `return_mode='ref'` so each call adds ≤80 tokens to agent context regardless of response size.

## 2. Concrete API signature and data shape

```python
from typing import Literal, Optional

@dataclass
class DispatchResult:
    success: bool                       # always present
    engine_used: str                    # e.g., "swarm/kimi"
    dispatch_id: str                    # UUID for later retrieval
    fallback_chain: list[str]           # engines tried
    tokens_used: int
    cost_usd: float
    # The following fields are mode-dependent:
    text: Optional[str] = None          # full text, only present in 'full' mode
    summary: Optional[str] = None       # ~200‑char summary, only present in 'summary' mode

    def fetch_text(self) -> str:
        """Retrieve full response from harness store (async or sync).
        Raises if response no longer available."""
        ...

    def fetch_summary(self) -> str:
        """Retrieve AI‑generated summary from harness store."""
        ...

def dispatch(
    prompt: str,
    engine: str | list[str] | None = None,
    *,
    return_mode: Literal["ref", "summary", "full"] = "ref",
    **kwargs
) -> DispatchResult:
    """
    Send prompt to harness engines.  The default 'ref' mode returns only
    metadata (dispatch_id, tokens_used, cost).  No text or summary is loaded
    into the agent’s context.  Use .fetch_text() or .fetch_summary() on the
    returned object to retrieve the content when needed – this places the
    result string under the agent’s explicit control.

    Modes:
      'ref'      – text=None, summary=None.
      'summary'  – summary is populated (~200 chars), text=None.
      'full'     – text is populated (legacy behaviour).
    """
```

## 3. Default that maximises context preservation

**`return_mode='ref'`** – the returned `DispatchResult` is ~80 bytes (success, engine, dispatch_id, tokens, cost). The agent can build logic to decide later whether to read the full output. For 30 dispatches, the agent context grows by ~2 400 tokens vs. ~750 000 tokens if `full` were used (assuming 25 K per response). This is the maximum offload: the agent never touches the response text unless it explicitly invokes `fetch_text()`.

## 4. Escape hatch when the agent needs the full response

- **Immediate full text**: call `dispatch(..., return_mode='full')` – returns `DispatchResult` with `.text` populated (same as current API).
- **Deferred access**: use the default `ref` mode, then later call `result.fetch_text()` or `result.fetch_summary()`. The agent can read the returned string into a variable (and thus into context) only when it decides the response is critical.
- **Summarised on-demand**: `result.fetch_summary()` returns a short (≤200 char) AI‑condensed version, which the agent may put into context with minimal token cost.

## 5. One risk specific to this lens

**Retrieval latency / availability** – the harness must retain dispatches (text + summary) long enough for the agent to call `fetch_text()`. If the agent runs in a long session and may fetch old results hours later, the store must be persistent across time. If the store drops entries (e.g., after 10 minutes), calls to `fetch_text()` will fail, and the agent may be forced to re‑dispatch, wasting tokens. The design must guarantee that all dispatch results are kept for at least the session’s lifetime (or until explicitly purged by the agent). This requires a durability contract between the harness and the store (Redis / S3 with TTL tied to session ID).

## C4-cost-aware-routing  (deepseek/deepseek-v4-flash)

## COST-AWARE ROUTING + TELEMETRY

### 1. Headline recommendation

Make `dispatch()` return a lightweight `DispatchReceipt` by default, and expose real-time per-session telemetry via `budget_status()` so the agent can monitor offload ratio and enforce a hard budget ceiling.

### 2. Concrete numbers / API signature / data shape

```python
# Default dispatch – returns minimal receipt (no full text)
from harness import dispatch, budget_status

receipt: DispatchReceipt = dispatch(
    prompt="summarize …",
    return_full=False,          # default
    engine="auto",              # cheapest-first router
    budget_usd=0.20,            # optional per-call cap
    max_latency=10.0            # seconds, fallback if cheapest too slow
)

# DispatchReceipt fields
class DispatchReceipt(NamedTuple):
    dispatch_id: str            # "abc-123"
    engine: str                 # "deepseek-v4-flash"
    tokens_in: int              # 20_000
    tokens_out: int             # 22_000
    cost_usd: float             # 0.0003
    latency_ms: float           # 1200
    # no .text, no error, no fallback chain

# Retrieving full text only when needed (escape hatch)
full_result: DispatchResult = dispatch(
    prompt="summarize …",
    return_full=True,           # opt‑in to full text in context
)

# Or later, from a receipt:
from harness import get_dispatch_text
text: str = get_dispatch_text(receipt.dispatch_id)  # burns context

# Budget & offload telemetry
status: dict = budget_status()
# {
#     "session_tokens_total": 1500000,
#     "session_cost_total": 0.14,
#     "offload_ratio": 0.85,            # harness / total tokens
#     "remaining_budget_usd": 5.00,     # user‑set ceiling
#     "dispatches_fired": 18,
#     "engines_used": {
#         "deepseek-v4-flash": 12,
#         "kimi": 5,
#         "mimo": 1
#     },
#     "avg_cost_per_token": 9.3e-8
# }
```

### 3. The default that maximizes context preservation

**`return_full=False`** and **`engine="auto"`** (cheapest-first routing).  
The receipt is ~200 bytes per dispatch – negligible compared to full text (5–50 KB). Over 30 dispatches, context growth from receipts ≈ 6 KB vs 750 KB+ for full responses.

The cheapest-first router uses a cost-per-token table (updated every dispatch) and current `budget_status()["remaining_budget_usd"]` to select the engine with lowest marginal cost that meets latency tolerances. If budget is exhausted, it falls back to the free engine (MiMo) or halts.

### 4. The escape hatch

Two explicit opt‑ins to get full text without paying the context tax for the common case:

- **Per-dispatch**: pass `return_full=True` in the call.  
  The agent consciously decides to ingest the response immediately.

- **Deferred retrieval**: call `harness.get_dispatch_text(id)` after the fact.  
  This allows the agent to decide *after* reviewing the receipt whether the response is worth its context budget.

Both burn context only when the agent judges it necessary.

### 5. One risk / failure mode

**Telemetry feedback loop consumes the agent’s context**.  
If the agent calls `budget_status()` after every dispatch, the returned dict (even if small) accumulates. A runaway loop of 100 dispatches + 100 telemetry calls adds ~20 KB. The design should encourage polling at strategic points (e.g., after a batch) and allow the agent to pass a `reset=True` flag to clear accumulated local telemetry state. Without this, the context‑preservation win is partially eroded.

## C5-agent-sdk-patterns  (mimo/mimo-v2.5-pro)

1. **Headline:** Adopt a structured `ToolResult` envelope with `summary` (default-returned) and `content_ref` (fetch-on-demand), mirroring what every major SDK has converged on—the tool returns a compact object, not raw output.

2. **Concrete API / data shape:**

```python
@dataclass
class DispatchResult:
    # --- ALWAYS returned (context cost: ~200-400 tokens) ---
    success: bool
    engine_used: str
    dispatch_id: str
    summary: str                  # ≤500 tokens, agent-authored or auto-distilled
    tokens_used: TokenLedger      # .in_, .out, .cost_usd (flat ints)
    truncated: bool               # True means full text exists, not returned

    # --- NEVER in-band unless requested ---
    content_ref: str | None       # e.g. "harness://dispatch/abc-123"
    
    def full(self) -> str:        # network round-trip, returns raw .text
        return _fetch_content(self.content_ref)

    def chunks(self, n: int = 3) -> list[str]:
        return _fetch_content_chunks(self.content_ref, n)
```

This maps directly to OpenAI's tool-call schema (`tool_call_id` + compact result string), Anthropic's `tool_result` block (structured content, not freeform), and LangGraph's `ToolMessage` (which also caps by default and requires explicit `return_direct` for full dump). Aider truncates tool output at a configurable `--tool-output-token-budget` (default 4K). Nobody ships a tool that silently returns 50K tokens.

3. **The default that maximizes context preservation:** The default `summary` field—auto-generated by the harness via a cheap pass (rule-based: first 30 lines + line count + token count, or via a single DeepSeek-flash call at ~$0.001). The agent's context sees only this. The `summary` is the return value the LLM "sees" as the tool result. This is exactly what LangGraph does with `handle_tool_error` summarization and what Anthropic's Claude does internally when it truncates overly long tool outputs—the harness just makes it explicit.

4. **The escape hatch:** `result.full()` is a synchronous fetch that costs a network round-trip but *not* context pre-allocation. The agent calls it only when the summary contains an actionable signal (e.g., `truncated=True` and the agent needs test output to debug). Alternatively, `result.chunks(n)` pages through large output n-thousand-token segments at a time, letting the agent implement its own RAG-style retrieval. The key invariant: **the agent never ingests what it didn't explicitly request**. This mirrors OpenAI's `file_search` tool (returns citations, not documents) and Anthropic's MCP resource pattern (return a URI, not the content).

5. **One risk / failure mode:** **Summary fidelity.** If the auto-summary drops the one error line that mattered, the agent will dispatch again, wasting cycles in a wrong-direction loop. Every SDK that summarizes tool output faces this—Claude Code's own file-read tool truncates but preserves line numbers so the agent can request the missing range. Mitigation: the `summary` must always include the **last N lines** (tail) in addition to the head, and the `DispatchResult` must include `exit_code` or `error_excerpt` as a top-level field so the agent can detect "needs full output" from metadata alone, without ever touching the body.
